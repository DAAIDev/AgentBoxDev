The point of the reviewer agent. When Chris reviews a PR today, he asks three questions:
  1. Is there anything structurally or security-wise broken?
  2. Is it properly feature-flagged?
  3. Is it going to break anything else?
The reviewer agent answers exactly those three, with evidence, before the PR is opened. If it can't answer all three with "PASS," the PR doesn't open. That's the bar.

3. The triage worker WEDGE 1

3.1 Where it lives

New file: AgentBoxDev/triage.mjs. Same Cloud Run service. Imported from server.js and fired after every successful upsertFeedbackTask.

3.2 Trigger

Two paths, redundant on purpose:

  1. Webhook-fired: in server.js at the existing /api/feedback-tasks/webhook handler, after upsertFeedbackTask, fire setImmediate(() => triageFeedbackTask(taskId)). Mirrors how pushKanbanTaskToGitHub is already wired.
  2. Safety poller: every 60s, claim and triage anything with planner_status='pending' AND created_at < NOW() - INTERVAL '1 minute'. Catches webhook-fired runs that crashed.

3.3 Code shape

// AgentBoxDev/triage.mjs
import Anthropic from '@anthropic-ai/sdk';
import { db } from './db.mjs';
import { ghSearchCode, ghGetFile } from './tools/github.mjs';
import { TRIAGE_SYSTEM_PROMPT } from './prompts/triage.mjs';

const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

const GH_SEARCH_CODE_TOOL = {
  name: 'gh_search_code',
  description: 'Search code in a DAAITeam repo. Returns matching paths and line snippets.',
  input_schema: {
    type: 'object',
    properties: {
      repo: { type: 'string', enum: ['CRMBackend', 'CRMFrontEnd'] },
      query: { type: 'string', description: 'Code search query, e.g. "function handleFeedback" or "@Controller portal"' },
    },
    required: ['repo', 'query'],
  },
};

const GH_GET_FILE_TOOL = {
  name: 'gh_get_file',
  description: 'Fetch the full content of a file from a DAAITeam repo at HEAD of dev branch.',
  input_schema: {
    type: 'object',
    properties: {
      repo: { type: 'string', enum: ['CRMBackend', 'CRMFrontEnd'] },
      path: { type: 'string' },
    },
    required: ['repo', 'path'],
  },
};

export async function triageFeedbackTask(taskId) {
  // Atomic claim — only one worker triages a row.
  const claimed = await db.query(
    `UPDATE mcp_feedback_tasks
        SET planner_status = 'running', planner_started_at = NOW()
      WHERE id = $1 AND planner_status = 'pending'
      RETURNING *`,
    [taskId],
  );
  if (claimed.rowCount === 0) return;
  const task = claimed.rows[0];

  try {
    const userPrompt = renderTaskPrompt(task);

    // Agentic loop: Claude calls tools, we execute, return result, repeat until stop.
    let messages = [{ role: 'user', content: userPrompt }];
    let response;
    for (let turn = 0; turn < 20; turn++) {
      response = await anthropic.messages.create({
        model: 'claude-sonnet-4-6',
        max_tokens: 8000,
        system: TRIAGE_SYSTEM_PROMPT,
        tools: [GH_SEARCH_CODE_TOOL, GH_GET_FILE_TOOL],
        messages,
      });
      messages.push({ role: 'assistant', content: response.content });

      if (response.stop_reason !== 'tool_use') break;

      const toolResults = await Promise.all(
        response.content
          .filter(b => b.type === 'tool_use')
          .map(async b => ({
            type: 'tool_result',
            tool_use_id: b.id,
            content: await runTool(b.name, b.input),
          })),
      );
      messages.push({ role: 'user', content: toolResults });
    }

    const plan = extractJsonOutput(response);
    await persistPlan(task, plan);
    if (plan.scope !== 'skip') {
      await openProjectedIssues(task, plan);
    }
  } catch (err) {
    await db.query(
      `UPDATE mcp_feedback_tasks
          SET planner_status='failed', planner_error=$2
        WHERE id=$1`,
      [taskId, String(err)],
    );
    throw err;
  }
}

async function runTool(name, input) {
  if (name === 'gh_search_code') return await ghSearchCode(input.repo, input.query);
  if (name === 'gh_get_file')    return await ghGetFile(input.repo, input.path);
  throw new Error(`Unknown tool: ${name}`);
}

The two new fields (v0.3.2):

  • synthesized_bug_md — agent's clean technical restatement of the bug. Names file/function/expected behavior, not the customer's words. Becomes the issue's lede; the implementer reads this, not the original report.
  • proposed_fix_md — agent's suggested shape for the fix, bulleted by file. Guidance, not mandate — implementer may deviate with documented reason. The reviewer's Check 3 compares actual diff against this shape as a second axis beyond "did the test pass?"

4. What gets opened in GitHub

Issues, not PRs. Triage produces a specification, not code. The PR comes after the implementer runs (Wedge 4).

Each projected issue is the agent's "spec" — it carries everything downstream stages need:

  • Title: [<TENANT>][<crm_task_id>] <original title> — grep-friendly
  • Hidden trailer: <!-- agentbox-sync: tenant=<slug> crm_task_id=<id> --> — machine-readable link back to DB row (this pattern already exists in kanban-github-sync.mjs)
  • Body: acceptance criteria, reproducer, pre-written test stub, paired-with link (if scope=both), planner reasoning, link to source kanban task
  • Labels: agent-eligible, tenant:<slug>, area:<X>, type:<bug|feature>, priority:<P>, paired (if scope=both), state:planned

If scope=skip, open ONE fallback issue with agent-skip + reason label, in the planner's best-guess repo. Surfaces to humans for manual triage.

5. GitHub Actions runner WEDGE 2

Each agent attempt runs in an ephemeral GitHub-hosted runner. Triggered by GitHub events — no orchestrator daemon, no custom VM image, no DevTemplate headless prep work for v1. Brandt's "spin up a VM with Playwright" idea is preserved; we just use GitHub's VMs instead of building our own.

5.1 The two workflows

Two workflow files. They fire on different events so the reviewer can run cleanly against just the pushed diff, and a failed reviewer doesn't have to rerun the implementer.

FileTriggerJob
.github/workflows/agent-implement.yml issues.labeled with agent-eligible Set up env → invoke implementer agent → push agent branch (does NOT open PR yet)
.github/workflows/agent-review.yml push to agent/** branches Set up env → invoke reviewer (3 checks) → on PASS open PR against dev; on FAIL comment + label issue

Each in-scope repo (CRMBackend, CRMFrontEnd) gets its own copy of both workflows. For paired work items the planner labels both issues; both workflows fire independently.

5.2 Per-attempt steps (agent-implement.yml)

  1. actions/checkout with ref: dev
  2. Setup Node 20, Postgres 18, Playwright + browsers (cached — ~30s steady-state, ~3min cold)
  3. Run repo-local ./bin/setup-ci.sh (headless DB + seed; thin wrapper, no DevTemplate dependency)
  4. Read the work item from the labeled issue body; extract acceptance criteria + test stub
  5. Create agent/<crm_task_id>-attempt-<n> from origin/dev
  6. Commit the pre-written test (with a guard hook that pins this file's hash)
  7. Invoke claude-code with the implementer prompt; let it iterate until acceptance test passes or max-turns hit
  8. git push origin agent/<crm_task_id>-attempt-<n> (just the agent branch — NOT a push to dev)
  9. actions/upload-artifact: transcript, screenshots, logs

5.3 Per-attempt steps (agent-review.yml)

  1. actions/checkout on the just-pushed agent branch
  2. Setup env + ./bin/setup-ci.sh
  3. Invoke reviewer agent (three checks — see §7)
  4. If all checks PASS: gh pr create --base dev; PR body = reviewer's structured report
  5. If any FAIL: gh issue comment with summary; add state:blocked-needs-human label to the source issue
  6. Upload Playwright videos + reviewer evidence as Action artifacts; link from PR body / issue comment

5.4 Why GitHub Actions over a custom VM (v1)

  • Zero infra to build. No Packer recipe, no image build pipeline, no Cloud Run dispatcher, no DevTemplate --agent mode.
  • Native event triggers. Label transitions on issues fire workflows automatically — no orchestrator daemon needed for v1.
  • Same isolation. Each Action run is a fresh ephemeral VM. One agent run can't see another's state.
  • Native PR + issue integration. gh is preinstalled and authenticated; comments, labels, PR creation all "just work."
  • Cost. At 20 runs/day × ~30 min = 600 min/day = ~$144/mo on private repos (free tier adds ~2000 min/mo on team plans). Cheaper and more elastic than a Cloud Run service that's mostly idle.
  • Self-hosted escape hatch. If the 60-min cap or cost becomes painful, point Actions at a single beefy self-hosted GCE runner — no rewrite, just a runs-on label change.

5.5 What we give up vs the v0.3 GCE plan

  • 60-min hard cap per run. Same as the VM TTL I'd set anyway. Wash.
  • No pre-baked image. Each run installs deps from cache (~30s warm, ~3min cold). Tolerable.
  • Per-tenant config requires runtime input. Workflow runs with whatever env you pass. For v1 (dev seed only) this is fine; for per-tenant runs later, pass --tenant as a workflow input.
  • DevTemplate headless work shelved. Still useful for local dev and human onboarding, just not blocking the agent loop. Decouple and ship at its own pace.

5.6 Why Playwright matters

The reviewer runs the acceptance test and a "headed" render of the same test that captures a video via Playwright's video: 'on'. The video uploads as an Action artifact and is linked from the PR body. Chris (or any reviewer) can watch the 30-second clip before approving. That's what makes the reviewer credible — not a JSON verdict, but observable behavior.

6. Implementer agent WEDGE 4

6.1 Job

Read the work item (acceptance criteria + test stub from issue). Iterate on code in one or both in-scope repos under additive-only API discipline. Stop when the pre-written test passes and the existing suite is green.

6.2 Hard rules (enforced as commit-time guards, not just prompted)

RuleEnforcement
Cannot modify the pre-written test fileGit pre-commit hook checks tests/e2e/feedback-<crm_task_id>.spec.ts is unchanged from initial commit. Hard fail.
Cannot push to main, dev, or any protected branchPush only allowed to agent/* branches. Branch protection enforced at GitHub.
Cannot force-pushForce-push disabled at GitHub branch protection level.
Cannot make breaking API changesStatic check: routes, DTOs, status codes diffed against origin/dev. See §8 of v0.2 spec.
Cannot run destructive DB opsTool layer refuses DROP, TRUNCATE on non-test tables.
Cannot access tenant prod DBsNo Cloud SQL proxy installed on runner image. Network allow-list excludes prod CIDRs.

6.3 Stopping conditions

  • Acceptance test passes + existing suite green → success → hand to reviewer
  • Hit max_turns (default 40) → fail; orchestrator retries
  • Emits BLOCKED: needs-migration / needs-coordinated-deploy / needs-tenant-data → orchestrator escalates to human

6.4 Tool access

  • Filesystem (read/edit/write/grep) on workspace
  • Shell: npm, npx, git diff/status, curl localhost only
  • Playwright via codegen and direct API
  • gh issue view for re-reading context
  • NO general network egress; NO direct API calls outside the allow-list

7. Reviewer agent — replaces Chris's manual PR review WEDGE 5

This stage is the whole point. When this works reliably, Chris's PR review burden drops to "watch the video + skim the diff" instead of "read every line and check three different things." It's the wedge that justifies the rest of the build.

7.1 The three checks (one agent, one report)

Check 1 — Security & structural review

Mechanism
Invoke Claude Code with the /security-review skill against the agent branch. That skill is Anthropic's, battle-tested, scoped exactly to "what's wrong here."
Scans for
Secrets in code; SQL injection; missing auth guards on new endpoints; XSS; broken access control (e.g. tenant data leak); unsafe deserialization; SSRF; XXE; prototype pollution
Plus structural
God-objects; missing error handling on critical paths (DB writes, external API calls); unawaited promises; new endpoints without input validation
Output
{ status: 'pass' | 'fail', findings: [{ severity, file, line, description }] }
Fail criterion
Any finding of severity critical or high

Check 2 — Feature-flag gating

Mechanism
Static analysis first, Sonnet only for ambiguous cases
Step 1 — Static
Walk the diff. For every new file or function in tenant-specific areas (matched against per-repo .claude/REVIEW_RULES.md), check whether it's reachable only behind an ENABLE_* environment-variable check or a TenantFeatures table lookup.
Step 2 — Sonnet
For new code in mixed/ambiguous areas, ask: "Is this change company-specific or generic? If company-specific, where is the flag?" Constrained Sonnet call, ~2k tokens.
Per-repo rules
Lives at CRMBackend/.claude/REVIEW_RULES.md and CRMFrontEnd/.claude/REVIEW_RULES.md. Lists which areas are company-specific (portal, billing UI, ticket views) and which are generic (auth, RBAC, core CRUD).
Output
{ status, new_code_paths: [{ file, line, gated_by, verdict }] }
Fail criterion
Any tenant-specific code path without a flag — see project memory: "Company-specific features must be feature-flagged"

Check 3 — Regression / blast-radius / additive-only

Mechanism
Mostly mechanical, some Sonnet judgment for adjacency picks
Step 1
Run new acceptance test (headless + headed with video). Required pass.
Step 2
Run full existing suite in each touched repo. Required pass.
Step 3
Sonnet picks N (default 5) "adjacent" test files based on import graph + path proximity to changed files. Run them. Required pass.
Step 4
Sonnet picks 3-5 adjacent Playwright user flows. Smoke them. Soft signal (warn, don't fail).
Step 5 — Additive-only check
Static diff of route registry, DTO schemas, status codes, Prisma schema vs origin/dev. Any rename / removal / semantic change fails. (See v0.2 spec §8.)
Step 6 — Diff resembles proposed shape (v0.3.2)
Compare the actual diff against the triage agent's proposed_fix_md. Did the implementer touch the files the planner expected? Did it apply the kind of change described? Big deviation without documented reason in the workpad = surface to human (warn, don't fail outright — the implementer might have found a genuinely better path). Catches "test passes but fix is structurally unrelated to the actual bug" — i.e. test-hacking, mock injection, hardcoded values.
Output
{ status, acceptance_test, existing_suite, adjacent_tests, additive_check, fix_shape_match }
Fail criterion
Required steps fail; additive-only violation fails; fix_shape_match=diverged without workpad rationale → warn (Chris reviews)

7.3 Failure routing

Check failedAction

# Reviewer Agent System Prompt

You are the reviewer for an agent-opened PR. You run three checks:
security/structural, feature-flag gating, and regression/blast-radius.
Your output is the PR body verbatim — write it with a human reader
in mind (Chris).

Invoke the `/security-review` skill. Collect all findings. Mark

Run, in order:
1. The new acceptance test (headless, then headed with video).
2. The full existing suite in each touched repo.
3. 5 adjacent test files (you pick — by import graph + path).
4. 3-5 adjacent Playwright flows (you pick — by feature proximity).
5. Additive-only check: confirm no existing route, DTO field,
   status code, or Prisma column was renamed, removed, or had
   semantic change.

Markdown document matching the template in §7.2 of the build plan.

- Changing a status code on an existing endpoint
- Reordering required parameters
- Changing a response field's type or nullability
